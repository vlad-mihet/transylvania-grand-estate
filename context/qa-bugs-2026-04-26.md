## Third-pass review additions (2026-04-26 ultrathink)

After the second pass I asked myself "is this actually production-ready?" Discovered five new gaps and addressed them:

### 1. CRITICAL — `qa-reset.ts` could reset prod admin passwords **[FIXED]**

`apps/api/prisma/qa-reset.ts` shipped with the api package and would reset all admin passwords to `QaPass123!` if executed. Even if normal deploys don't run it, defense in depth matters. Added `if (process.env.NODE_ENV === 'production') process.exit(1)` at the top. Verified: refuses with NODE_ENV=production, runs in dev.

### 2. RISK — Refresh-token replay window **[FIXED]**

`/auth/refresh` issued a new pair but didn't invalidate the just-used refresh token. Stolen-token reuse window = 7 days (the JWT TTL).

Fix: in `apps/api/src/auth/auth.controller.ts:64-71` and `apps/api/src/auth/auth.service.ts:78-100`, the refresh handler now revokes the previous jti before issuing the new pair. Single-use rotation enforced. Verified live:
- RT1 → first refresh succeeds, returns RT2
- RT1 replay → 401 "Refresh token has been revoked"
- RT2 → succeeds, returns RT3

### 3. CORRECTED — Lesson XSS over-reported **[VERIFIED-SAFE]**

Earlier security audit flagged `<ReactMarkdown>` on lesson content as XSS-vulnerable. Re-audit:
- `react-markdown` v9 escapes raw HTML by default; no `rehype-raw` configured. `<script>` becomes literal text.
- Default URL transformer strips `javascript:` from links.
- `videoUrl` is canonicalized to youtube-nocookie / player.vimeo via `lessonEmbedUrlSchema` (`packages/types/src/schemas/_primitives.ts:131-203`).
- `<LessonVideoPlayer>` further sandboxes the iframe (`sandbox="allow-scripts allow-same-origin allow-presentation"`, no top-navigation, no popups).

The lesson rendering is **safe by default**. Original finding was overstated.

### 4. UX — Flash-of-form before redirect **[FIXED]**

The `useEffect`-based permission guards I added in passes 1 and 2 caused a brief flash of the form (and triggered data-fetch via DetailPageShell) before the redirect fired for users without permission. EDITOR visiting `/properties/[id]/edit` would see the populated form for ~100ms before /403.

Fix: added `if (!can("X")) return null;` after the useEffect on all 18 guarded pages plus `if (isAgent) return null;` on the dashboard. The `null` short-circuits before DetailPageShell fires its query and before the JSX renders. The redirect (already scheduled in the useEffect) takes the user to /403 immediately, no flash.

### 5. Comprehensive list/detail page audit **[CLEAN]**

Audited 14 list/detail pages I'd previously only scanned. All `<Can>`-gated correctly:
- list pages: cities, testimonials, bank-rates, articles, financial-indicators
- detail pages: agents/[id], developers/[id], cities/[id], testimonials/[id], bank-rates/[id], academy/courses/[id]
- academy: students/, students/[id], invitations/

Every Create/Edit/Delete/Toggle is either `<Can>`-wrapped or uses `permissions={...}` prop on `RowActions`/`EntityDeleteButton`.
