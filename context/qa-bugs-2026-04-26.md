### P1-017 — Command palette still routes AGENT to admin-only paths → /403 **[FIXED]**

- **Discovered by:** browser walkthrough A2 — AGENT opens command palette (`Ctrl+K`), clicks "Inquiries", lands on `/403`. Sidebar nav to "My Inquiries" works as expected.
- **Root cause:** Phase-1 P1-002 was marked "collapsed into P0-001 fix" because P0-001 widened `AGENT_ALLOWED_PREFIXES` to include `/properties/` and `/inquiries/` — but the trailing slashes in those prefixes only match detail/edit deep links (`/properties/abc`, `/inquiries/xyz`). The bare admin list paths the palette links to (`/properties`, `/inquiries`, `/developers`, `/cities`) don't match any prefix → `isPathAllowedForAgent` returns false → AuthGuard sends AGENT to `/403`. The palette's permission filter (`COMMAND_ACTIONS.filter(a => !a.requires || can(a.requires))`) doesn't catch this because AGENT genuinely has the `read` perm — just no UI surface that consumes the bare list page.
- **Fix:** lifted `AGENT_ALLOWED_PREFIXES` and `isPathAllowedForAgent` into a new shared module `apps/admin/src/lib/agent-paths.ts`. Single source of truth. AuthGuard imports from there (`auth-guard.tsx`). The palette also imports and adds a second-pass filter: `if (role === "AGENT" && !isPathAllowedForAgent(action.href)) return false`. AGENT-specific entries (My Listings / My Inquiries / Profile) remain visible because their hrefs (`/my-listings`, `/my-inquiries`, `/profile`) ARE in the allowlist.
- **Why this slipped Phase 1:** the Phase-1 review of P1-002 noted "if you widen `AGENT_ALLOWED_PREFIXES`, this collapses into the same fix" — but the chosen widening (trailing slashes for deep-link compatibility) didn't cover the bare list paths. The static audit signed off on the P0-001 widening without re-walking the palette in a real session. Wave 3 browser pass caught it.
- **Phase-2 regression spec idea:** can't be e2e (palette is browser-only). Best mitigation: a unit test on `isPathAllowedForAgent` asserting that every bare admin path that has an AGENT-allowed companion (`/properties` ↔ `/my-listings`, `/inquiries` ↔ `/my-inquiries`) returns false; and that the palette filter applied to a fixture AGENT user excludes those palette entries by id. Lower priority than UI test.
- **Source-of-truth files:** `apps/admin/src/lib/agent-paths.ts` (new), `apps/admin/src/components/auth/auth-guard.tsx` (now imports from shared), `apps/admin/src/components/command-palette/command-palette.tsx` (filters by reachability).

### P0-016 — `@tge/types` runtime stale-dist masks every schema fix **[FIXED — Wave 3.5]**

- **Discovered by:** P1-013 fix (added `.nullable()` to `developerId`/`agentId`) didn't take effect at API runtime even though src + typecheck were clean. Same payload kept returning `Validation failed: developerId expected string, received null`.
- **Root cause:** `packages/types/package.json` declares `"exports": { "./*": { "node": "./dist/*.js", "default": "./src/*.ts" } }`. Node honors `node`, never falls through to `default`. Compiled `dist/schemas/property.js` was last modified **Apr 23** (4 days stale vs my Apr 27 src edit). `apps/api/tsconfig.json` paths map `@tge/types/*` → src for **typecheck only**; at Node runtime tsconfig paths are ignored. Result: every schema source edit since Apr 23 has been silently ignored at runtime until someone manually ran `pnpm --filter @tge/types build`.
- **Scope of vulnerability** (audit, this session): only `@tge/types` and `@tge/data` ship a `dist/` and have the `node→dist` exports pattern. All other `@tge/*` packages (`api-client`, `utils`, `hooks`, `branding`, `i18n`, `ui`, `tailwind-config`) export src directly — no dist, no risk. The 4 Next apps are unaffected because they consume packages via `transpilePackages` (webpack reads src).
- **Why this slipped Phase 1:** the e2e harness builds dist as a CI step (`.github/workflows/ci.yml:44-45,90-91`) before running tests, and the production Dockerfile builds it before `nest build`. Local dev was the only blind spot — and it was the one we hit.
- **Fix (Wave 3.5):**
  - `packages/types/package.json` + `packages/data/package.json` — added `"watch": "tsc -p tsconfig.json -w --preserveWatchOutput"`.
  - `apps/api/package.json` — `dev` script now runs `concurrently -k -n types,data,api -c blue,magenta,green "pnpm --filter @tge/types watch" "pnpm --filter @tge/data watch" "nest start --watch"`. dist stays fresh on every src save; Nest's own watcher picks up the rebuilt dist and restarts. Added `concurrently` as a devDependency.
  - `apps/api/package.json` — added `"prebuild"` step that runs `pnpm --filter @tge/types build && pnpm --filter @tge/data build` before `nest build` for prod parity.
  - `README.md` "Build pipeline notes" — extended to document the auto-watch behaviour.
- **Verification (live):** rebuilt dist + bounced API; ADMIN PATCH `/properties/<id>` with `{developerId: null, price: 275999}` returned 200, row updated, developerId now null. Then bounced again on the new orchestrator (`pnpm --filter @tge/api dev`); same PATCH still 200; orchestrator log shows `[types]`, `[data]`, `[api]` prefixed output from the three concurrent watchers.
- **Phase-2 regression spec idea:** can't be e2e (the bug is purely about dev-environment loading, not API contract). Best mitigation: a CI smoke that asserts `git status -- packages/{types,data}/dist` is clean after running `pnpm --filter @tge/{types,data} build`, so a developer who edits src but forgets to commit the rebuilt dist gets caught. But since CI itself rebuilds dist, this is largely belt-and-suspenders. **Recommended deferred** unless the issue resurfaces.
- **Alternatives considered + rejected:**
  - Flipping `exports.node` to point at src — creates a parity gap between dev (loads src) and prod (loads dist via Dockerfile build); a build-pipeline regression would only show up in prod.
  - Dropping dist entirely — would break the API's prisma seed and Dockerfile contract.
  - Manual runbook only — has been tried (README already documented the build step). It's how we got here.

### P0-015 — Login → immediate /login bounce: StrictMode double-fire races single-use RT rotation **[FIXED]**

- **Discovered by:** browser walkthrough — AGENT logs in, navigates to /my-listings, instantly bounces back to /login. Smoking gun in API log: two `/auth/refresh` POSTs arrive 6ms apart at 23:14:28; first returns 201, second returns 401 (`Refresh token has been revoked`).
- **Root cause:** React StrictMode (Next.js dev) mounts every client component twice. `AuthProvider`'s `useEffect` calls `restoreSession()` on each mount, so two parallel `/api/auth/refresh` requests fire after login. With Wave-1's single-use rotation (`auth.service.ts:78-95`), the first call succeeds and revokes the original jti; the second call hits the denylist and returns 401. The admin BFF route at `apps/admin/src/app/api/auth/refresh/route.ts:34` then deletes the `refreshToken` cookie on any 401 — clobbering the freshly-rotated RT the first call just set. Next render: `AuthGuard` sees `!user`, redirects to `/login`. Production wouldn't hit this (no StrictMode double-fire), but every dev session would.
- **Fix (two parts):**
  - `auth-provider.tsx`: hoisted the refresh call into a module-level `inflightRestore` Promise lock so concurrent calls share one network request. Survives StrictMode mount/unmount/mount because module state outlives component instances. Released on next tick so a genuine later refresh can fire.
  - `app/api/auth/refresh/route.ts`: stopped deleting the cookie on 401. A 401 can mean "this jti was just rotated by a parallel call"; clobbering the cookie kills a live session. If the RT is genuinely expired, the next call fails again and the user re-logs — same UX, no silent session-loss.
- **Why this slipped Phase 1:** the e2e harness exercises rotation via supertest in a single process — no React, no StrictMode. The bug only surfaces in a real browser running in dev mode. The new `refresh-token-single-use.e2e-spec.ts` (Wave 1) still passes; this is purely the client-side reaction to a 401.
- **Phase-2 regression spec idea:** can't be e2e (browser-only). Best mitigation: a unit test on the new module-level `restoreSession` helper that fires it 3× in parallel and asserts only one fetch went out. Lower priority than UI test but easy to author.
- **Source-of-truth files:** `apps/admin/src/components/auth/auth-provider.tsx` (module-level lock), `apps/admin/src/app/api/auth/refresh/route.ts` (cookie-preserve on 401).

### P0-014 — admin BFF can't reach API: stale `:3333` in `apps/admin/.env` **[FIXED]**

- **Discovered by:** browser walkthrough — login screen returned "Auth service unavailable" after Wave 2 forced `PORT=4000` in `apps/api/.env`.
- **Root cause:** `apps/admin/.env` had `API_URL=http://localhost:3333/api/v1` and `NEXT_PUBLIC_API_URL=http://localhost:3333/api/v1`. The Phase-1 bug log called the port mismatch out for the API side ("Dev port mismatch — admin BFF can't reach it") and Wave 2 fixed it by forcing the API onto :4000. The admin's outbound URL was the other half of the same mismatch and was missed in Wave 2.

Same handler covers all auth proxies (`refresh/route.ts` etc.), so every credential-bearing flow would have failed identically.

Best mitigation: an `apps/admin/.env.example` synced with `apps/api/.env.example` so both move together, plus a smoke note in `README.md` ("Build pipeline notes" already documents the API ports — extend with admin's API_URL).

### P1-013 — `developerId`/`agentId` reject `null` on PATCH; UI's "None" option breaks save **[FIXED]**

- **Discovered by:** browser walkthrough, immediately after P1-012 fix. AGENT clicked Save again; API returned 400 `Validation failed` with `developerId: expected string, received null`.
- **Root cause:** `packages/types/src/schemas/property.ts:71-72` declared `developerId: z.string().uuid().optional()` and `agentId: z.string().uuid().optional()` — accepts string or undefined, NOT null. The admin form's "None" option (`__none__` in the Select) maps to `null` in form state to mean "detach the relation". On submit, `toPropertyPayload` passes `null` through; API rejects.
- **Why it didn't block AGENT before:** the AGENT field-sanitizer (`apps/api/src/properties/properties.service.ts:33-89`) drops `developerId`/`agentId` from the effective payload before service writes — but validation fires *before* the service ever sees the request, so AGENT users got the same 400 as ADMIN.
- **Fix:** schema now declares `.nullable().optional()` for both fields. Service-side update logic at `properties.service.ts:791-800` already handles `null` correctly (treats as `disconnect: true`), so no service change needed. The "None" option now persists as a real "detach" on update — both for ADMIN choosing None and for AGENT submitting an unchanged null inherited from the row's prior state.
- **Phase-2 regression spec idea:** extend `properties-agent-sanitizer.e2e-spec.ts` with a case asserting AGENT PATCH `{developerId: null}` returns 200 and the row keeps its prior developerId (sanitizer drops the field). And a new ADMIN case asserting PATCH `{developerId: null}` succeeds and the row's developerId is now null.
- **Source-of-truth files:** `packages/types/src/schemas/property.ts:71-72`. Both apps consume via tsconfig paths to `packages/types/src/`, no rebuild needed.

### P1-012 — Numeric form inputs not coerced; submit silently fails with focus-loop **[FIXED]**

- **Discovered by:** browser walkthrough (AGENT edits `/properties/<id>/edit`, changes price by €1, clicks Save → focus bounces to price input on every click; no network request fires).
- **Root cause:** `<Input type="number" {...form.register("price")} />` keeps the form-state value as a *string* (the raw event target). Form schemas use plain `z.number()` (intentionally — the v4 upgrade dropped `z.coerce.number()` so `useForm<T>` infers `number`, not `unknown`). On submit, zodResolver rejects the string, react-hook-form's default `shouldFocusError: true` jumps focus to the offending field, the user clicks Save again, repeat.
- **Affected forms:**
  - `apps/admin/src/components/forms/property-form.tsx` — 10 numeric inputs (price, latitude, longitude, bedrooms, bathrooms, area, landArea, floors, yearBuilt, garage)
  - `apps/admin/src/components/forms/developer-form.tsx` — `projectCount`
  - `apps/admin/src/components/forms/city-form.tsx` — `propertyCount`
  - `apps/admin/src/components/forms/bank-rate-form.tsx` — `rate`, `sortOrder`, `maxLtv`, `maxTermYears`, `processingFee`, `insuranceRate`
  - Already clean: `course-form.tsx:251`, `lesson-form.tsx:120,184` (use `valueAsNumber: true` / `setValueAs`).
- **Fix:** added `register(field, { valueAsNumber: true })` for required number fields and `register(field, { setValueAs: (v) => v === "" || v == null ? null : Number(v) })` for `.optional().nullable()` / `.optional()` fields. Helpers extracted at the top of each form for grep-ability. Admin tsc clean.
- **Why this slipped Phase 1:** the static + API-contract review used the API matrix, not browser submit. Forms only break at the React/Zod boundary; the API never sees a malformed payload, so no audit-log row, no API 4xx, nothing observable from outside the browser.
- **Phase-2 regression spec idea:** can't be e2e (the bug is purely client-side). Either (a) Playwright UI test if/when we adopt one, or (b) a unit test on each form: render → setValue numeric field via fireEvent → form.handleSubmit → assert `data.price === 275000` (number, not string). Lower priority than UI test.
- **Source-of-truth files:** the 4 form files above. Each now uses the same `requiredNumber` / `optionalNullableNumber` / `optionalNumber` helper shape so future numeric fields get the right pattern by copy-paste.

## Third-pass review additions (2026-04-26 ultrathink)

After the second pass I asked myself "is this actually production-ready?" Discovered five new gaps and addressed them:

### 1. CRITICAL — `qa-reset.ts` could reset prod admin passwords **[FIXED]**

`apps/api/prisma/qa-reset.ts` shipped with the api package and would reset all admin passwords to `QaPass123!` if executed. Even if normal deploys don't run it, defense in depth matters. ### 2. RISK — Refresh-token replay window **[FIXED]**

`/auth/refresh` issued a new pair but didn't invalidate the just-used refresh token. Stolen-token reuse window = 7 days (the JWT TTL).

Fix: in `apps/api/src/auth/auth.controller.ts:64-71` and `apps/api/src/auth/auth.service.ts:78-100`, the refresh handler now revokes the previous jti before issuing the new pair. Single-use rotation enforced. Verified live:
- RT1 → first refresh succeeds, returns RT2
- RT1 replay → 401 "Refresh token has been revoked"
- RT2 → succeeds, returns RT3

### 3. CORRECTED — Lesson XSS over-reported **[VERIFIED-SAFE]**

Earlier security audit flagged `<ReactMarkdown>` on lesson content as XSS-vulnerable. Re-audit:
- `react-markdown` v9 escapes raw HTML by default; no `rehype-raw` configured. `<script>` becomes literal text.
- Default URL transformer strips `javascript:` from links.
- `videoUrl` is canonicalized to youtube-nocookie / player.vimeo via `lessonEmbedUrlSchema` (`packages/types/src/schemas/_primitives.ts:131-203`).
- `<LessonVideoPlayer>` further sandboxes the iframe (`sandbox="allow-scripts allow-same-origin allow-presentation"`, no top-navigation, no popups).

The lesson rendering is **safe by default**. Original finding was overstated.

### 4. UX — Flash-of-form before redirect **[FIXED]**
