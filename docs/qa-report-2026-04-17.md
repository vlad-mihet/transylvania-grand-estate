## New blockers discovered in re-verification

### 🛑 Blocker #14 — Admin is entirely down because `proxy.ts` exports the wrong name

- **App:** admin.
- **Where:** `apps/admin/src/proxy.ts`.
- **What:** The file was renamed from `middleware.ts` (correct fix for Minor #10) but the export was not renamed. It still reads:

  ```ts
  export function middleware(req: NextRequest) { ... }
  ```

  Next.js 16 `proxy.ts` requires `export function proxy(req)` or `export default`. Result: every admin request (login, dashboard, any CRUD page) returns HTTP 500 with:

  ```
  The file "./src\proxy.ts" must export a function, either as a default export
  or as a named "proxy" export.
  ```

- **Caveat on detection:** the very first admin request after a cold start can succeed before Next fully discovers `proxy.ts`. Once warmed, every subsequent request 500s. This is why the bug didn't surface in the first pass — I hit `/en/login` once, got 200, and moved on. The re-run suite caught it because it makes many sequential admin requests.
- **Fix:** rename the function:

  ```ts
  export function proxy(req: NextRequest) { ... }   // renamed from `middleware`
  ```

  Or simpler: `export default function proxy(req) { ... }`.

- **Severity rationale:** Admin is the entire internal product surface. Zero functionality while this is broken.

### 🛑 Blocker #15 — Reveria `/en/blog` and `/ro/blog` return HTTP 500

- **App:** reveria — only the blog *listing* route. Individual article detail pages (`/en/blog/ghid-impozite-proprietari-romania`) return 200 fine.
- **Reproducible:** yes, every run of the suite and every manual curl.
- **What the API sees:** `GET /api/v1/articles?status=published&limit=24` with `X-Site: REVERIA` returns 200 with 8 articles (7 `guide`, 1 `market-report`). All have `coverImage`, `publishedAt`, localized `title`/`excerpt`. The failure is on the Reveria render side.
- **HTML clue:** the 500 response body contains `<meta name="next-error" content="not-found"/>` — suggesting a `notFound()` was thrown from inside the listing render, but the framework mapped it to 500 instead of rendering `not-found.tsx`. The blog listing page itself does not call `notFound()` (see `apps/reveria/src/app/[locale]/blog/page.tsx`), so the throw is coming from a descendant component — likely `ArticleCard`, `BlogCategoryTabs`, or one of the shared imports (`@tge/i18n/navigation`, `@tge/ui`).
- **Verified parity with neighbors:**

  ```
  curl -I http://localhost:3002/en/blog                     → 500
  curl -I http://localhost:3002/en/blog?category=guide      → 500
  curl -I http://localhost:3002/ro/blog                     → 500
  curl -I http://localhost:3002/en/blog/ghid-impozite-...   → 200   (detail works)
  ```

- **How to diagnose:** open the Reveria dev-server terminal and hit `/en/blog` — the stack trace should name the component and line. Candidate areas: `ArticleCard`'s use of `t('categories.${article.category}')` (both `guide` and `market-report` have translations in en/ro/fr/de, so not that); `ArticleCard`'s `localize(article.title, locale)` if any article has an un-localized title; the async `useTranslations` + `useLocale` in a Server Component boundary (Reveria's `ArticleCard` is `"use client"` — correct).
- **Quick sanity check:** remove the `<ArticleCard>` render entirely and confirm the page renders. If yes, the bug is in ArticleCard. If no, it's in BlogCategoryTabs or PageHeader.
- **Severity rationale:** both `/en/blog` and `/ro/blog` are 500; the blog is linked from the Reveria nav.
